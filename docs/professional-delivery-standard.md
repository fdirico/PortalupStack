# Professional Delivery Standard

## Objective

PortalUP Stack Codex must help build client-grade applications, not quick demos. Agents must optimize for durable architecture, maintainable code, strong validation, modern UI, and honest professional judgment.

## Non-negotiables

- Do not implement ambiguous product, architecture, security, or UX decisions as guesses.
- Ask for missing information when it changes architecture, user experience, security, data model, or delivery risk.
- Challenge requests that would produce weak, outdated, unsafe, unmaintainable, or inaccessible software.
- Prefer permanent fixes over cosmetic patches.
- Validate with real commands, code inspection, tests, static checks, screenshots, logs, or reproducible evidence when available.
- Separate facts, assumptions, hypotheses, decisions, and open questions.
- Do not claim readiness without evidence.

## Build Quality

Professional development must cover:

- Requirements and acceptance criteria.
- Architecture and module boundaries.
- Data flow, contracts, errors, and observability.
- Security, roles, permissions, and secrets.
- Tests for happy path, failure path, edge cases, and regression.
- Performance, accessibility, responsiveness, and operational behavior.
- Rollback and support model when release risk exists.

## UI Standard

PortalUP applications should feel modern, dense, and professional:

- Medium or small typography for operational tools.
- Compact buttons, optimized menus, clear hierarchy, and efficient layouts.
- Responsive behavior designed explicitly for desktop and mobile.
- Modern charts, states, sliders, filters, tabs, data grids, and useful visual emphasis.
- 3D or rich visual elements only when they improve comprehension, brand, or interaction.
- No outdated, oversized, low-density, or generic admin-looking screens.

## Professional Pushback

When the user request is under-specified, the agent must choose one of:

- Ask targeted questions before implementing.
- State assumptions and proceed only if the risk is low.
- Propose a better approach and explain why the requested one is weaker.
- Refuse or pause if the request risks production, data, security, or client trust.
