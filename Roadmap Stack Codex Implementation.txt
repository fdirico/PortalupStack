Roadmap arquitectónico: PortalUP Stack basado en GStack para Codex
0. Visión del producto
Nombre interno sugerido
PortalUP Stack for Codex

O más corto:

pstack-codex
Objetivo

Crear una adaptación de GStack para que funcione sobre Codex, reutilizando la mayor cantidad posible de:

roles
prompts
comandos
checklists
flujos de revisión
flujos de QA
flujos de documentación
flujos de shipping

Pero reemplazando lo que dependa de Claude Code por mecanismos compatibles con Codex:

CLAUDE.md       → AGENTS.md
.claude/skills  → .agents/skills
slash commands  → skills invocables con $skill o comandos wrapper
Claude hooks    → scripts / wrappers / workflows
Claude-specific → Codex-compatible instructions

AGENTS.md es un formato abierto pensado como un “README para agentes”, usado para dar instrucciones, comandos de setup, pruebas, estilo de código y contexto del proyecto a diferentes agentes de programación.

1. Principio de arquitectura

Yo no haría un fork sucio de GStack. Haría una arquitectura en 4 capas:

┌─────────────────────────────────────────────┐
│ PortalUP Stack                              │
│ Branding, reglas PortalUP, casos FileNet,   │
│ RPA, BPM, SQL, AWS, propuestas comerciales  │
└─────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────┐
│ Codex Adapter Layer                         │
│ AGENTS.md, .agents/skills, scripts, plugins │
└─────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────┐
│ GStack Methodology Layer                    │
│ roles, comandos, QA, review, ship, docs     │
└─────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────┐
│ Codex Runtime                               │
│ Codex CLI / IDE / App                       │
└─────────────────────────────────────────────┘

Esto evita quedar amarrados a Claude Code y permite que mañana también pueda existir:

PortalUP Stack for Claude
PortalUP Stack for Cursor
PortalUP Stack for Gemini CLI
PortalUP Stack for OpenCode

5. Fase 2 — Diseño del adaptador GStack → Codex
Objetivo

Definir las reglas de traducción.

Mapeo principal
GStack / Claude Code            PortalUP Stack / Codex
---------------------------------------------------------
CLAUDE.md                       AGENTS.md
.claude/skills/gstack           .agents/skills/
slash command /review           skill portalup-review
slash command /qa               skill portalup-qa
slash command /ship             skill portalup-ship
Claude-specific instruction     Codex-compatible instruction
Claude hooks                    script opcional / workflow wrapper

Codex permite guardar skills en .agents/skills, $HOME/.agents/skills y otros scopes; también puede detectar cambios en skills, y cada skill debe tener un SKILL.md con name y description.

Decisión clave

No intentar replicar los slash commands exactamente al inicio.

En Codex lo natural sería usar:

$portalup-review
$portalup-qa
$portalup-ship
$portalup-filenet-review

Luego, si queremos una experiencia más parecida a GStack, se puede crear un wrapper:

pstack review
pstack qa
pstack ship
pstack filenet-review
Entregables
docs/decisiones-arquitectura/ADR-002-estructura-skills.md
docs/decisiones-arquitectura/ADR-003-no-fork-directo.md
scripts/validate-skills.js
Criterio de aceptación

Se aprueba cuando tengamos una regla clara para convertir cualquier skill de GStack a skill Codex.

8. Fase 5 — Instalador local
Objetivo

Que el stack no sea solo una carpeta manual, sino algo instalable.

Instalación inicial simple

Para Linux/macOS:

git clone https://github.com/portalup/portalup-stack-codex.git
cd portalup-stack-codex
./scripts/install-local.sh

Para Windows:

git clone https://github.com/portalup/portalup-stack-codex.git
cd portalup-stack-codex
.\scripts\install-local.ps1
Qué debe hacer el instalador
validar que existe Codex
crear carpeta ~/.agents/skills si no existe
copiar o enlazar skills
copiar templates
validar SKILL.md
mostrar skills instaladas
Entregables
scripts/install-local.sh
scripts/install-local.ps1
scripts/validate-skills.js
docs/instalacion-codex.md
Criterio de aceptación

Un equipo PortalUP debe poder instalarlo y probar:

$portalup-review
$portalup-qa
$portalup-propuesta-comercial

Sin modificar archivos manualmente.

11. Fase 8 — Empaquetado como plugin Codex

Codex diferencia entre skills como formato de autoría y plugins como unidad instalable/distribuible; la documentación oficial indica que los plugins pueden agrupar varias skills y opcionalmente app mappings, configuración MCP y assets.

Objetivo

Convertir portalup-stack-codex en algo distribuible internamente.

Resultado deseado
PortalUP Stack Plugin

Incluye:

skills
templates
assets
scripts
configuración opcional
Criterio de aceptación

Instalable por cualquier desarrollador/consultor PortalUP sin copiar carpetas manualmente.

14. Definition of Done

Una skill no se considera lista hasta cumplir esto:

Tiene SKILL.md válido
Tiene name y description
Tiene instrucciones claras
Tiene formato de salida
Tiene criterios de severidad
Tiene ejemplos de uso
No depende de Claude Code
Funciona con Codex
Fue probada con mínimo 2 casos
Tiene score mínimo 75/100
Está documentada en README
15. Riesgos del proyecto
Riesgo 1 — Portar demasiado

Problema: querer copiar todo GStack desde el día uno.

Mitigación: MVP con 8 a 12 skills máximo.

Riesgo 2 — Skills demasiado largas

Codex usa progressive disclosure, pero la descripción inicial de cada skill es clave para que pueda seleccionarla bien; si hay muchas skills, las descripciones pueden acortarse para no saturar el contexto.

Mitigación: descriptions cortas, específicas y con palabras gatillo.

Riesgo 3 — Dependencias Claude-only

Problema: algunos comandos pueden depender de funciones específicas de Claude Code.

Mitigación: clasificarlos como:

port directo
port adaptado
reemplazar
excluir
Riesgo 4 — Resultado bonito pero poco accionable

Problema: la IA entrega texto elegante, pero no operativo.

Mitigación: cada skill debe forzar salidas con:

acciones
comandos
validaciones
riesgos
criterios de aceptación
Riesgo 5 — Falta de identidad PortalUP

Problema: terminar con un clon genérico de GStack.

Mitigación: agregar capa PortalUP desde la semana 3:

FileNet
RPA
BPM
SQL
AWS
propuestas comerciales
soporte técnico

18. Mi recomendación como arquitecto

Yo no haría esto:

copiar GStack completo
cambiar nombres
probar a ver si funciona

Haría esto:

1. Fork conceptual de GStack
2. Adaptador limpio para Codex
3. Skills núcleo
4. Skills PortalUP
5. Testing con casos reales
6. Instalador
7. Documentación
8. CLI opcional
9. Plugin distribuible

La decisión arquitectónica más importante sería esta:

PortalUP Stack no debe ser un clon de GStack. Debe ser una distribución compatible con Codex, inspirada y parcialmente derivada de GStack, pero especializada en los servicios reales de PortalUP.

Con ese enfoque, no estarías reinventando lo inventado. Estarías haciendo una adaptación profesional, reutilizable y con identidad propia.
